# Architecture

**Analysis Date:** 2026-03-26

## Pattern Overview

**Overall:** Workspace-root planning repo wrapped around a nested Node.js CLI/package that installs and orchestrates markdown-defined AI workflows.

**Key Characteristics:**
- The main implementation lives in `get-shit-done/`, not in the workspace root.
- User-facing behavior is mostly data-driven: commands, workflows, agents, templates, and references are markdown files, while Node `.cjs` modules provide filesystem, parsing, validation, and state mutation primitives.
- Runtime installs are multi-target: the installer in `get-shit-done/bin/install.js` can materialize assets into `~/.claude`, `~/.codex`, `~/.gemini`, `.github`, and other runtime-specific homes.
- Project memory is file-based. Planning and codebase-map artifacts live under `.planning/`, and the CLI reads and mutates those files directly instead of using a database or service process.
- The repository also contains exported n8n workflow definitions at `wf-latest.json` and `wf-discovery.json`, but those JSON workflows are adjacent artifacts rather than the control plane for the GSD package itself.

## Layers

**Workspace Layer:**
- Purpose: Holds planning state, local runtime installs, and exported workflow artifacts for this working repo.
- Key locations: `.planning/`, `.claude/`, `.codex/`, `wf-latest.json`, `wf-discovery.json`.
- Depends on: The packaged source in `get-shit-done/` for the implementation model.
- Used by: Local agent sessions and planning workflows.

**Packaging and Install Layer:**
- Purpose: Publishable npm package surface and runtime installation logic.
- Key locations: `get-shit-done/package.json`, `get-shit-done/bin/install.js`.
- Contains: CLI entry registration, runtime selection, config-dir resolution, hook/agent/command installation, and cross-runtime conversion rules.
- Depends on: Source assets in `get-shit-done/commands/`, `get-shit-done/agents/`, `get-shit-done/get-shit-done/`, and `get-shit-done/hooks/`.
- Used by: `npx get-shit-done-cc`, package consumers, and tests that validate installation behavior.

**Command Definition Layer:**
- Purpose: Define the user-visible slash-command or skill surface.
- Key locations: `get-shit-done/commands/gsd/*.md`.
- Contains: 57 command specs with frontmatter and prompt bodies, for example `get-shit-done/commands/gsd/map-codebase.md` and `get-shit-done/commands/gsd/execute-phase.md`.
- Depends on: Workflow docs, agent docs, and the `gsd-tools` CLI for atomic operations.
- Used by: Claude, Codex, Copilot, Cursor, Windsurf, and other runtime installs generated by the installer.

**Workflow and Knowledge Layer:**
- Purpose: Encode reusable orchestration logic and supporting reference material.
- Key locations: `get-shit-done/get-shit-done/workflows/*.md`, `get-shit-done/get-shit-done/references/*.md`, `get-shit-done/get-shit-done/templates/**/*.md`.
- Contains: 56 workflows, reusable decision/reference docs, and templates for `.planning/` artifacts.
- Depends on: The CLI utility layer for concrete reads, writes, validation, and path resolution.
- Used by: Command definitions and specialized agents.

**Agent Definition Layer:**
- Purpose: Describe specialized subagents and their allowed tools/process.
- Key locations: `get-shit-done/agents/*.md`.
- Contains: Role-specific instructions such as `get-shit-done/agents/gsd-planner.md`, `get-shit-done/agents/gsd-executor.md`, and `get-shit-done/agents/gsd-codebase-mapper.md`.
- Depends on: Templates, references, workflows, and `.planning/` context files.
- Used by: Workflow orchestrators that fan out work by concern or phase.

**CLI Utility Layer:**
- Purpose: Provide the executable implementation behind the markdown workflows.
- Key locations: `get-shit-done/get-shit-done/bin/gsd-tools.cjs`, `get-shit-done/get-shit-done/bin/lib/*.cjs`.
- Contains: Routing, path discovery, config loading, roadmap parsing, phase math, state updates, verification, template filling, workstream support, and security helpers.
- Depends on: Node built-ins plus repository files on disk.
- Used by: Commands, workflows, tests, and agents that need structured project data.

**Hook and Guard Layer:**
- Purpose: Add runtime checks and context instrumentation around agent tool use.
- Key locations: `get-shit-done/hooks/*.js`, with publishable copies built into `get-shit-done/hooks/dist/` by `get-shit-done/scripts/build-hooks.js`.
- Contains: Update checks, prompt-injection warnings, workflow guards, context monitoring, and statusline output.
- Depends on: Runtime-provided stdin/tool-event payloads and local filesystem state.
- Used by: Installed runtime environments after `install.js` wires them into config.

**Test and CI Layer:**
- Purpose: Lock down installer behavior, CLI utilities, hook correctness, and workflow conventions.
- Key locations: `get-shit-done/tests/*.test.cjs`, `get-shit-done/scripts/run-tests.cjs`, `get-shit-done/.github/workflows/test.yml`, `get-shit-done/.github/workflows/security-scan.yml`.
- Contains: Node test runner entry, focused test files for config, roadmap, workstreams, install conversions, security scans, and more.
- Depends on: The shipped package code and fixture-style repo state.
- Used by: Local `npm test` and GitHub Actions.

## Data Flow

**Install and Materialize Runtime Assets:**

1. A user runs the npm binary exposed by `get-shit-done/package.json`, which points to `get-shit-done/bin/install.js`.
2. `get-shit-done/bin/install.js` parses flags such as `--claude`, `--codex`, `--all`, `--global`, and runtime-specific config-dir overrides.
3. The installer selects source assets from `get-shit-done/commands/`, `get-shit-done/agents/`, `get-shit-done/get-shit-done/`, and `get-shit-done/hooks/dist/`.
4. It rewrites paths and tool mappings as needed for each target runtime, then writes the generated install into folders like `.claude/` or `.codex/`.
5. Those installed folders become the execution surface for an agent host, while the nested package remains the source of truth in this repo.

**Command Execution and Planning Mutation:**

1. A runtime command or skill resolves to a markdown command definition under `get-shit-done/commands/gsd/`.
2. That command points to a workflow in `get-shit-done/get-shit-done/workflows/` and usually delegates file operations to `get-shit-done/get-shit-done/bin/gsd-tools.cjs`.
3. `gsd-tools.cjs` routes the request into domain modules like `lib/init.cjs`, `lib/state.cjs`, `lib/phase.cjs`, `lib/roadmap.cjs`, and `lib/verify.cjs`.
4. Those modules discover the correct `.planning/` root, read markdown/JSON state, compute phase or workstream context, and write updated artifacts back to disk.
5. The workflow may then spawn one or more agents from `get-shit-done/agents/` to create or verify docs, code, or summaries.

**Codebase Mapping Flow:**

1. The `map-codebase` command definition in `get-shit-done/commands/gsd/map-codebase.md` routes to the workflow in `get-shit-done/get-shit-done/workflows/map-codebase.md`.
2. That workflow uses `get-shit-done/agents/gsd-codebase-mapper.md` and `gsd-tools.cjs init map-codebase` to load repo context.
3. Mapper agents inspect the live repository and write focused outputs into `.planning/codebase/`, including `ARCHITECTURE.md`, `STRUCTURE.md`, `STACK.md`, and adjacent documents.
4. Later planning and execution commands consume those files as brownfield context.

**Test Flow:**

1. `npm test` in `get-shit-done/` runs `node scripts/run-tests.cjs`.
2. `get-shit-done/scripts/run-tests.cjs` enumerates every `*.test.cjs` file in `get-shit-done/tests/`.
3. Node’s built-in test runner executes the files, covering installer logic, CLI library behavior, hook validation, and workflow invariants.
4. CI repeats that path through GitHub workflow files in `get-shit-done/.github/workflows/`.

**State Management:**
- Durable state is primarily file-based under `.planning/`.
- Runtime install state is filesystem-based under hidden runtime directories such as `.claude/` and `.codex/`.
- The CLI utility layer treats the filesystem as the API surface: it resolves project roots, reads markdown and JSON, and writes back normalized artifacts.

## Key Abstractions

**Runtime Target:**
- Purpose: A host environment the package can install into and adapt for.
- Examples: Claude, Codex, Gemini, Copilot, Cursor, Windsurf, Antigravity, and OpenCode in `get-shit-done/bin/install.js`.
- Pattern: Installer-driven adapter table with runtime-specific directories, tool-name mappings, and config resolution.

**Command / Workflow Pair:**
- Purpose: Split user-facing intent from reusable orchestration logic.
- Examples: `get-shit-done/commands/gsd/map-codebase.md` paired with `get-shit-done/get-shit-done/workflows/map-codebase.md`.
- Pattern: Thin command shell delegating to a richer workflow document.

**Agent Spec:**
- Purpose: A markdown-defined specialized worker contract.
- Examples: `get-shit-done/agents/gsd-executor.md`, `get-shit-done/agents/gsd-plan-checker.md`.
- Pattern: Prompt/config document with explicit responsibilities and tool allowances.

**Planning Artifact:**
- Purpose: Human-readable project memory and handoff state.
- Examples: `.planning/STATE.md`, `.planning/ROADMAP.md`, `.planning/codebase/ARCHITECTURE.md`.
- Pattern: Markdown and JSON files mutated by CLI helpers instead of in-memory services.

**CLI Domain Module:**
- Purpose: A focused implementation module behind the `gsd-tools` router.
- Examples: `get-shit-done/get-shit-done/bin/lib/state.cjs`, `get-shit-done/get-shit-done/bin/lib/template.cjs`, `get-shit-done/get-shit-done/bin/lib/workstream.cjs`.
- Pattern: Functional CommonJS modules imported by the central dispatcher.

## Entry Points

**Package Binary:**
- Location: `get-shit-done/bin/install.js`
- Trigger: Running `get-shit-done-cc` via npm/npx.
- Responsibilities: Runtime selection, install/uninstall, path conversion, and asset materialization.

**CLI Utility Router:**
- Location: `get-shit-done/get-shit-done/bin/gsd-tools.cjs`
- Trigger: Invoked by workflows, scripts, or tests.
- Responsibilities: Parse atomic commands and route them into domain modules.

**Workflow Commands:**
- Location: `get-shit-done/commands/gsd/*.md`
- Trigger: Slash command or skill execution inside an installed runtime.
- Responsibilities: Choose workflows, delegate to agents, and describe outputs.

**Hooks:**
- Location: `get-shit-done/hooks/*.js`
- Trigger: Runtime lifecycle events like session start or tool use.
- Responsibilities: Advisory guards, context monitoring, and status reporting.

**n8n Exports:**
- Location: `wf-latest.json` and `wf-discovery.json`
- Trigger: Imported into n8n rather than executed by Node in this repo.
- Responsibilities: Define automation DAGs for the adjacent workflow artifacts stored at the workspace root.

## Error Handling

**Strategy:** Fail fast for installer/CLI misuse, but prefer graceful degradation for runtime hooks and exploratory helpers.

**Patterns:**
- `get-shit-done/get-shit-done/bin/lib/core.cjs` exposes `error()` to print a message and exit the process for invalid CLI usage.
- Path and config loading helpers in `lib/core.cjs` and `lib/init.cjs` use conservative fallbacks when optional files are missing.
- Hooks in `get-shit-done/hooks/*.js` are intentionally defensive and often exit quietly to avoid breaking the host runtime on malformed input.
- `get-shit-done/scripts/build-hooks.js` validates hook syntax before copying files into `hooks/dist/`, preventing broken releases from being published.
- Tests cover edge cases like runtime conversion, config migration, path replacement, and security scanning so failures surface before publish time.

## Cross-Cutting Concerns

**Filesystem Safety:**
- `get-shit-done/get-shit-done/bin/lib/security.cjs` and path helpers in `lib/core.cjs` protect against invalid workstream names, unsafe paths, and malformed input.

**Cross-Runtime Compatibility:**
- The installer centralizes runtime differences so the same command, workflow, agent, and hook assets can be repackaged for multiple hosts.

**Documentation-as-Code:**
- The repo treats markdown as executable configuration. Commands, workflows, agents, references, and templates are all first-class program inputs.

**Verification:**
- Verification exists at several layers: CLI validators in `lib/verify.cjs`, hook syntax validation in `scripts/build-hooks.js`, and broad regression coverage in `get-shit-done/tests/`.
