<objective>
Add stuck-processing auto-cleanup and pessimistic status lock to wf-latest so overlapping 3-minute trigger runs cannot grab the same lead and stuck leads recover automatically.

Purpose: The race condition is the most severe bug -- two overlapping runs read the same pending leads before either writes a status update, causing duplicate API spend and corrupted NocoDB state. The cleanup ensures no lead stays stuck in processing forever (threshold: 10 minutes).

Output: Modified wf-latest.json with cleanup node at trigger start and two-step fetch-then-lock pattern.
</objective>

const projectId = 'pb7f1zou786xyqc';

const token = 'SqqM8YcDuzXGijg0oY7PvqTcPX5H-r2sYqbAOWTN';

const tenMinutesAgo = new Date(Date.now() - 10 * 60 * 1000).toISOString();

const response = await $http.request({ method: 'GET', url: getUrl, headers: { 'xc-token': token } });
const stuckRows = response.list || [];

Replace the current single-step lead fetching (Read All Rows -> Filter Unprocessed -> enrichment chain) with a two-step pessimistic lock pattern:

Step 1 -- Fetch pending lead IDs only:
Create/modify the existing GET node to fetch only IDs with a strict limit of 5:
```
GET /api/v1/db/data/noco/pb7f1zou786xyqc/mey3zgihq7o4at9
  ?where=(status,eq,pending)
  &limit=5
  &fields=Id
```
Name this node "Get Pending IDs".

Step 2 -- Immediately lock all 5 leads:
After getting IDs, PATCH each row to `status=processing` BEFORE doing anything else. This is the critical race condition fix -- once a row is `processing`, no other trigger run will grab it.

Step 3 -- Fetch full records:
After locking, fetch the full record for each lead by ID.

The existing `Filter Unprocessed` Code node (id: 485fff1d) currently does `unique.slice(0, 5)` -- this logic moves to the API query (`LIMIT 5` + `WHERE status=pending`). The Code node can be removed or replaced.

New node chain after Cleanup Stuck Processing:
```
Cleanup Stuck Processing
  |
Get Pending IDs (HTTP GET ?where=(status,eq,pending)&limit=5&fields=Id)
  |
Parse Pending IDs (Code node: extract list, return items or {_skip:true} if empty)
  |
IF No Pending (IF node: check _skip flag -> NoOp if true)
  |
Loop Over Items [Batch Size: 1] (splitInBatches node)
  | (loop body)
  |
PATCH Status Processing (HTTP PATCH /.../{Id} body: {"status":"processing"})
  |
GET Full Record (HTTP GET /.../{Id})
  |
[... existing enrichment chain nodes, re-wired to receive current item from loop ...]
  |
  back to Loop Over Items input
  | (done output)
[END / NoOp]
```

Key implementation details:
- The Loop Over Items node wraps the ENTIRE enrichment chain (PATCH processing -> GET full record -> all enrichment nodes -> final NocoDB status update). Per D-08, batch size = 1.
- All existing `.first()` references in the enrichment chain MUST be updated to use `$json.fieldName` (current loop item context). Confirmed affected nodes from RESEARCH.md:
  - `OpenRouter - Clean Name`: change `$('Filter Unprocessed').first().json.company_name` to `$json.company_name`
  - `Serper - Search`: change `$('OpenRouter - Clean Name').first().json...` to reference the current loop item
  - `OpenRouter - URL Dedup`: change `.first()` references similarly
  - All NocoDB PATCH nodes: change `$('Filter Unprocessed').item.json.Id` to `$json.Id`
- The "done" output of Loop Over Items connects to a NoOp/end node (fires after all 5 leads processed)
- The "loop" output (main output) connects to the PATCH Status Processing node

NOTE: This task combines the status lock (FIX-01) with the batch loop setup (laying groundwork for FIX-02). The `.first()` reference cleanup is essential because the loop changes how items flow. However, the OpenRouter backoff (FIX-03) is NOT yet added -- that comes in Plan 03.
